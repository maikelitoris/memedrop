## 0.2 Core Mechanic Shift - Critical Context

Phases 1-3 had a stash/dismiss tension as the central mechanic: you opened a drop, saw the meme, and chose whether to STASH it or let it go forever. That scarcity model was intentional and described in Phase 1.

**PHASE 4 CHANGED THIS INTENTIONALLY. The shift is:**

**The core mechanic is now an INCREMENTAL COLLECTION system.** Memes auto-stash on every drop. The player's progression is driven by collecting duplicates to level cards up (LV 1-5), unlocking lore segments, and eventually earning Meme Coins from max-level dupes. The tension is no longer "keep or discard" - it is "what level is my collection at, and what lore have I unlocked?" This is the intended design.

Consequence: any spec or bug report that treats auto-stash as a regression is WRONG. Auto-stash is correct. The "STASH TO GALLERY" button must NOT be re-introduced. The post-reveal screen shows the card, confirms what was received (new card vs. level-up vs. max-level coin award), and offers CONTINUE.

# 2\. Phase 5 - Fix Specifications

Implement in the exact order listed. Do not start a step until the previous passes flutter analyze with zero warnings.

## Fix F-01: Reel Result Synchronisation (BUG-01)

Root cause: the result rarity is determined independently from the reel. They must be tied together at generation time.

**Correct flow in DropService / AssetService:**

- STEP 1 - Before showing the reel, determine the result: call selectRandomMeme() → get MemeData with a definite rarity tier.
- STEP 2 - Pass that rarity into generateReel(resultRarity). The winning tile at position \[31\] is set to resultRarity. The reel is built AROUND the result, never the other way around.
- STEP 3 - After reel lands, the card reveal uses the SAME MemeData object that was selected in Step 1. No second random call.

Updated generateReel function:

_function generateReel(resultRarity: RarityTier): RarityTier\[\] - identical to Phase 4 §6.2 but called with the already-known result. The result rarity at index \[31\] is a parameter, not a re-roll._

## Fix F-02: Reel Duration and Deceleration (BUG-05)

The reel feels too quick. The spin must feel weighty - like a real slot machine running out of momentum.

- Minimum total spin duration: 4.5 seconds (was approximately 2s).
- Animation curve: cubic-bezier(0.02, 0.85, 0.2, 1.0) - very slow start, peak speed around 30% of duration, long deceleration tail for the final 50%.
- Speed phases: fast scroll (0-40% of duration), constant fast (40-60%), gradual deceleration (60-100%).
- In Flutter: use an AnimationController with duration: Duration(milliseconds: 4500) and a CurvedAnimation with a custom Cubic curve. Drive the reel track translateX from an AnimationController listener.
- The final tile must come to a complete stop with no bounce or overshoot - the deceleration is the drama.

## Fix F-03: "?" Back Face on Reel Tiles (BUG-06)

Reel tiles represent the backs of cards flying past - you cannot see what they are. Each tile must show a card-back design.

- Each reel tile: dark background (#1A1A1A), a large "?" character centred (fontSize: 32sp, color: #333333, fontWeight: w300), and a 2px border in the rarity colour of that tile with a matching glow.
- The rarity border+glow on each tile still reflects its rarity colour - this is the only rarity signal visible during the spin.
- This creates the sensation of card backs rushing past - the colour of the border hints at rarity without revealing the card face.
- The winning tile at rest: shows a "?" briefly, then the card flip animation begins (see F-07).

## Fix F-04: Restore MemeCard3D - 3D Flip Widget (BUG-02)

The MemeCard3D widget was deleted by the Phase 4 implementer. This was a mistake. It must be fully restored from the Phase 3 spec.

**Core implementation (pure Matrix4, no 3D packages):**

- AnimationController \_flipController: duration 600ms.
- Front face: card image (or placeholder fill). Shows when \_flipController.value < 0.5.
- Back face: lore content (see F-06). Shows when \_flipController.value >= 0.5.
- Transform uses Matrix4.identity()..setEntry(3,2,0.001)..rotateY(angle) where angle goes 0 → pi (front→back) or pi → 2\*pi (back→front).
- At value == 0.5 (edge-on): switch which face is rendered. The back face is drawn with a horizontal mirror (scaleX: -1) so text reads correctly.
- Tap gesture on the card triggers \_flipController.forward() or \_flipController.reverse() depending on current state.
- Snap-back physics: if the user drags but releases before 50% flip, snap back to nearest face with a SpringSimulation (stiffness: 300, damping: 25).

Wire it into both:

- Reveal screen (post-reel card): MemeCard3D renders here. Tap to flip = show lore back. Initial state: back face (card is face-down). Auto-flip to front face as part of the landing animation sequence.
- MemeCardScreen (from gallery tap): MemeCard3D renders here. Tap to flip = toggle front/back.

## Fix F-05: Link MemeData to Cards - Data Wiring (BUG-04)

CollectionItem must carry a memeId that resolves to a MemeData entry in kMemeManifest. This was specified in Phase 3 but lost in Phase 4.

**Checklist:**

- CollectionItem model: ensure memeId: String field exists and is non-nullable.
- On drop open: selectRandomMeme() returns a MemeData. Store item.memeId = meme.id when creating CollectionItem.
- Everywhere a CollectionItem is used (gallery tile, card detail, history screen): look up MemeData with: final meme = kMemeManifest.firstWhere((m) => m.id == item.memeId, orElse: () => kFallbackMeme).
- kFallbackMeme: a placeholder MemeData with all fields set to safe defaults (empty strings, isPlaceholder: true). Never crash on a missing lookup.
- Gallery tile: use meme.name for tooltip or accessibility label. Placeholder tiles: show the rarity colour fill (current behaviour) - acceptable.
- Card front face: show meme.name bottom-left overlay (12sp, letterSpacing: 2, white, semi-transparent) when isPlaceholder is false.
- Card back face: use meme.origin, meme.lore, meme.peakEvent, meme.tags - see F-06.

**Locked segment treatment:**

- Wrap segment content in a Stack. Bottom layer: blurred text (ImageFilter.blur sigmaX:12, sigmaY:12 via BackdropFilter). Top layer: semi-transparent black overlay (#000000 at 60% opacity) + a padlock icon (Icons.lock_outline, size: 18, color: #444444) + "LV X TO UNLOCK" text (9sp, #555555).
- pointer-events: none on locked segments (AbsorbPointer wrapping the overlay).

**Unlock animation (triggered on level-up):**

- Blur dissolves: AnimationController 600ms, drives ImageFilter.blur sigmaX from 12 → 0.
- Overlay fades: overlay opacity 0.6 → 0.0.
- Gold shimmer sweep: a gradient Container (Color(0xFFFFD700).withOpacity(0.18) → transparent) translates left → right over 800ms on top of the segment.
- Padlock icon scales down and fades out (200ms, before blur starts).

Read unlock state from CardInstance.unlockedLoreSegments: a List&lt;int&gt; containing the indices (0-3) of unlocked segments. Index 0 = ORIGIN (LV 2), 1 = LORE (LV 3), 2 = PEAK MOMENT (LV 4), 3 = TAGS (LV 5).

# 3\. Phase 5 - New Feature Specifications

## Feature A: Card Edge Glow Effect - Rarity Halo (User Request)

When viewing a card full-screen (MemeCardScreen from gallery, or reveal screen post-reel), a gradient glow should bleed in from the screen edges - like the entire screen is backlit by the card's rarity colour. This is distinct from the card's own border glow.

**Implementation:**

- A full-screen Container placed behind the card (Stack, positioned: fill). Its decoration is a BoxDecoration with a RadialGradient (or four EdgeGradients via a CustomPainter).
- Preferred approach - CustomPainter that draws 4 edge gradients:
  - Top edge: rarity colour at opacity 0.18 → transparent, 120px tall.
  - Bottom edge: same.
  - Left edge: same, 80px wide.
  - Right edge: same.
- Rarity colour lookup: same as existing rarity colour map (Normie: #9E9E9E, Mid: #4FC3F7, Based: #AB47BC, Dank: #FF7043, Sigma: #FFD700).
- The glow breathes: AnimationController looping 3s, drives opacity between 0.12 and 0.22. This gives the card a living presence.
- On card flip (front → back): glow colour cross-fades with the same 600ms duration as the flip.
- Sigma cards: glow opacity is higher (0.20 → 0.35) to make them feel more mythical.

Widget: create lib/widgets/rarity_edge_glow.dart. Accepts rarity and child. Wraps child in a Stack with the glow painter behind.

## Feature B: Reveal Screen - Meme Name Typewriter

After the card settles face-up on the reveal screen, the meme's name appears below the rarity badge with a typewriter animation.

- Text: meme.name in ALL CAPS. Style: 14sp, letterSpacing: 2, color: white, fontWeight: w300.
- Animation: characters appear one-by-one using a Timer that reveals one character every 40ms, starting 400ms after card settles.
- Below name: meme.era (9sp, #666666). Fades in after name completes (300ms fade).
- These only show if meme.isPlaceholder == false.

- Do NOT reintroduce a "STASH TO GALLERY" button. Memes auto-stash. This is the Phase 4 mechanic shift and it is intentional.
- Do NOT add social features, push notifications, accounts, or network calls.
- Do NOT use any border-radius > 12px.
